单例模式：一个类只有一个实例

那么类的构造函数（默认构造，拷贝构造，移动构造），附值运算符重载，需要禁用或者私有化。



1.懒汉式单例：获取单例的函数中如果没有类，则创建初始化。

导致第一次创建的初始化的时间比较长。

DBConfig* DBConfig::config = nullptr;

DBConfig::getInstance(); //获取单例的函数中如果没有类，则创建初始化。

```c++
DBConfig* DBConfig::getInstance()
{
	if (config == nullptr)
	{
      config = new DBCofnig();
	}
  return config;
}
```

2.饿汉式单例：

DBConfig* DBConfig::config = new DBConfig(); //这里提前初始化

DBConfig::getInstance(); //获取单例的函数中只判断

```c++
DBConfig* DBConfig::getInstance()
{
  return config;
}
```



单例模式的多线程场景

```c++
DBConfig* DBConfig::getInstance()
{
  if(config == nullptr)
  {
    mut.lock();
    if (config == nullptr)
    {
        config = new DBCofnig();
      	//1.分配内存
      	//2.调用构造初始化
      	//3.赋值给config
				//如果config不用volaltile修饰，那么2和3的顺序可能会交换
    }
    mut.unlock();
  }
  return config;
}
volaltile DBConfig* DBConfig::config = nullptr; //volaltile修饰指针，防止对该指针操作进行重排序
```

编译器 / CPU 为了优化执行效率，会在**不改变单线程语义**的前提下，对无依赖的指令重排序。这里指令 2（初始化）和指令 3（赋值）没有数据依赖（CPU 看来只是给指针赋值，和初始化内存无关），因此可能被重排成：

```plaintext
1. 分配内存 → 3. 指针赋值 → 2. 调用构造函数
```





#### volatile 的真实作用

`volatile` 关键字的核心是：

1. 禁止编译器对该变量的读写做 “缓存优化”（每次读写都直接操作内存，而非寄存器）；
2. 禁止编译器对该变量的读写指令做**编译器层面的重排序**（但管不了 CPU 层面的重排序）；
3. 无法保证跨指令的顺序（比如无法约束 “构造函数执行” 和 “指针赋值” 的顺序）。

简单说：`volatile DBCofnig* config;` 只能保证 `config` 本身的读写不被缓存，但阻止不了 “构造函数（指令 2）” 和 “指针赋值（指令 3）” 的重排序。

那怎么阻止指令重排序？

#### 方案 1：使用原子操作（C++11 及以上，推荐）

`std::atomic` 能严格保证指令顺序，是多线程下最安全的方式：

```cpp
#include <atomic>

// 定义原子指针（核心：memory_order_seq_cst 保证全顺序）
std::atomic<DBCofnig*> config{nullptr};

// 线程 A：初始化
void initConfig() {
    DBCofnig* temp = new DBCofnig();  // 先在临时变量初始化，避免重排
    config.store(temp, std::memory_order_release);  // 发布语义：保证 temp 初始化完成后再赋值
}

// 线程 B：读取
void useConfig() {
    DBCofnig* ptr = config.load(std::memory_order_acquire);  // 获取语义：保证先读指针，再访问对象
    if (ptr != nullptr) {
        ptr->doSomething();
    }
}
```

- `memory_order_release`：保证该操作之前的所有指令（构造函数）都执行完毕，再执行指针赋值；
- `memory_order_acquire`：保证该操作之后的所有指令（访问对象），都在指针读取之后执行；
- 临时变量 `temp` 进一步规避了重排风险（先初始化完，再赋值给原子指针）。



#### 方案 2：使用互斥锁（兼容性好，简单易懂）

通过锁保证 “初始化” 和 “读取” 的互斥，本质是让线程 B 必须等线程 A 完成初始化后才能读取：

```cpp
#include <mutex>

DBCofnig* config = nullptr;
std::mutex config_mutex;

// 线程 A：初始化
void initConfig() {
    std::lock_guard<std::mutex> lock(config_mutex);
    if (config == nullptr) {  // 双重检查（可选，优化性能）
        config = new DBCofnig();
    }
}

// 线程 B：读取
void useConfig() {
    std::lock_guard<std::mutex> lock(config_mutex);
    if (config != nullptr) {
        config->doSomething();
    }
```

#### 方案 3：静态局部变量（单例场景，C++11 及以上）

如果 `config` 是单例对象，C++11 规定静态局部变量的初始化是线程安全的，且无重排风险：

```cpp
DBCofnig& getConfig() {
    static DBCofnig config;  // C++11 保证：初始化只执行一次，且线程安全
    return config;
}

// 线程 A/B 调用
getConfig().doSomething();
```

- 优点：零手动同步，编译器自动保证顺序和线程安全；
- 缺点：仅适用于单例场景，无法动态销毁 / 重新初始化。
