阻塞和非阻塞id是socket 文件描述符(fd)的两种工作状态，决定了依赖该 FD 的系统调用（如 `accept`/`recv`）是否会 “等待事件就绪”：

accept 是由监听fd工作状态决定的，如果是阻塞就会一直阻塞，直到收到通知。

listen_fd 的阻塞状态是预先设置的（默认阻塞），accept只是遵循这个状态的规则执行：

- 若 `listen_fd` 是阻塞的：`accept` 会阻塞到有可用连接为止，然后创建 `conn_fd` 并返回。
- 若 `listen_fd` 是非阻塞的：`accept` 发现没有可用连接时，立即返回错误（`EAGAIN`），不等待。

1. `accept` 等待的是 “`listen_fd` 上是否有已完成的连接”，这是 `listen_fd` 的事件。

### 四、总结：核心逻辑串联

1. **`socket` 是基础载体**：所有网络操作（`accept`/`recv`）依赖 socket FD，其阻塞 / 非阻塞状态决定系统调用行为。

2. **`accept` 的阻塞性由 `listen_fd` 决定**：默认阻塞，设置 `O_NONBLOCK` 后变为非阻塞，但非阻塞需配合 I/O 多路复用避免轮询。

3. `select`/`epoll` 是 “高效调度器”：
- 监控 `listen_fd` 的新连接事件，让 `accept` 按需调用（无需阻塞 / 轮询）；
   - 监控 `conn_fd` 的读写事件，支撑高并发通信；
- `epoll` 因无上限、高效率，成为高并发服务（如 Nginx、Redis）的首选。

简单说：**`socket` 是 “管道”，`accept` 是 “接管道的动作”，`select`/`epoll` 是 “同时看多个管道是否有动静的监控器”**，三者配合才能实现高效的网络服务
